What is Continuous Integration?

Continuous Integration is a software development practice where members of team integrate

their work frequently, usually each person integrates at least daily - leading to

multiple integrations per day. Each integration is verified by an automated build

(including test) to detect integration errors as quickly as possible. Many teams find that

this approach leads to significantly reduced integration problems and allows a team to

develop cohesive software more rapidly.

What is Continuous Delivery?

Continuous Delivery is a software development discipline where you build software in

such a way that the software can be released to production at any time.

You achieve continuous delivery by continuously integrating the software done by the

development team, building executables, and running automated tests on those

executables to detect problems. Furthermore, you push the executables into

increasingly production-like environments to ensure the software will work in

production. To do this you use a DeploymentPipeline.


To achieve continuous delivery you need:

  • a close, collaborative working relationship between everyone involved in delivery (often referred to as a DevOpsCulture[2]).

  • extensive automation of all possible parts of the delivery process, usually using a DeploymentPipeline

Continuous Delivery is sometimes confused with Continuous Deployment. Continuous

Deployment means that every change goes through the pipeline and automatically gets

put into production, resulting in many production deployments every day. Continuous

Delivery just means that you are able to do frequent deployments but may choose not

to do it, usually due to businesses preferring a slower rate of deployment. In order to

do Continuous Deployment you must be doing Continuous Delivery.

Your team can review the PRs and give feedback on changes. Reviewers can step

through the proposed changes, leave comments, and vote to approve or reject the PRs.

Depending on branch policies and other requirements, your PR might need to meet various

Everytime a Pull request is made to merge code onto the main branch for production, it

triggers workflows, a unit test is ran first to test code for errors. If there are no errors

it builds the image and push it to Dockerhub. Lastly, it deploys the app to Heroku.

How to set up project to use Github Actions?


To apply Github Actions you first have to go your project folder in your Pycharm editor.

  1. In the .github directory there is a workflows folder with three files: dev.yml, heroku_awake.yml, and prod.yml.
  2. In the dev.yml file you are going to edit where it says heroku_app_name: "type the name of your heroku app name"
  3. Below that line is heroku_email: "type your email address"
  4. In the heroku_awake.yml file in the jobs block of code the last line is says type the url of your heroku development app.
  5. and in the next block of where it says "keep_dev_awake" type the same thing your url heroku development app
  6. In the prod.yml you will do the same for dev.yml file type your heroku production app name and email.

If you look at the prod.yml file you will see on push to master branch when a push is done.

It will cause the jobs to work. One job is to run the test, one is to build the container to push to Dockerhub,

the other is to deploy the program to Heroku. When one of these events happen, push or pull, this trigger the rest

action to work.

Next, go to Github click on the setting tab.

When setting open you will see a list under General, scroll down until you see "Secrets"

Click on Secrets, a dropdown menu opens and you will see Actions, click on it. Actions secrets

opens up. To the right there is a button to create a New Repository secret.

You will create three repository secrets, one for DOCKER_USERNAME, DOCKER_PASSWORD, and HEROKU_API_KEY.

To generate a API_KEy you have to go to Heroku.com create an account. After creating an account login and go

click on account setting, then you will see Application tab. To create API token click the Create authorization

to generate a authorization token. This token is the value for the Heroku_API_KEY in the Github Actions

where you created your Docker username and Docker password.
